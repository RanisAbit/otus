# Инструкция по развертыванию сервера мониторинга

Данный документ описывает порядок развертывания сервера мониторинга инфраструктуры проекта.

В результате выполнения инструкции должны быть настроены:
- Prometheus в Docker контейнере;
- Grafana в Docker контейнере.

## Состав backend-серверов

Backend-серверы включают в себя:
- Docker для развертывания Grafana и Prometheus.


## Используемые файлы

### Скрипты
- `scripts/monitoring.sh` — установка  Grafana и Prometheus в Docker.

### Docker Compose
- `docker/grafana/docker-compose.yml`
- `docker/prometheus/docker-compose.yml`

### Конфигурационные файлы
- `configs/promteheus/prometheus.yml`
- `configs/sql-slave-server/mysqld.cnf`

## Предварительные требования

Перед началом развертывания необходимо убедиться, что:
- серверы `backend-1` и `backend-2` доступны по SSH;
- на серверах используется Ubuntu Server;
- серверы имеют доступ в интернет;
- известны IP-адреса backend-серверов;
- доступен пользователь `root` или пользователь с правами `sudo`.

Во всех примерах ниже предполагается выполнение команд от имени `root`.

## Таблица развертывания

| Сервер | Назначение | Используемые скрипты |
|---|---|---|
| `backend-1` | MySQL master, база данных WordPress, WordPress | `sql-master.sh`, `sql-wp.sh`, `wordpress.sh` |


## Общая схема запуска скриптов

Для каждого этапа используется одинаковый порядок действий:

1. Подключиться к нужному серверу по SSH.
2. Скачать необходимый скрипт из репозитория с помощью `curl`.
3. Сделать файл исполняемым.

```bash
chmod +x имя_скрипта.sh
```

4. Запустить скрипт.

```bash
./имя_скрипта.sh
```

Если скрипт не удалось скачать автоматически, его можно создать вручную и вставить в него содержимое из репозитория.

## Порядок развертывания

### Настройка сервера `backend-1`

1. Подключиться к серверу `backend-1`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/sql-master.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. Во время выполнения скрипта задать пароль для пользователя `repl`.
5. После завершения скрипта убедиться, что master-сервер MySQL запущен и отображает корректный статус.

### Настройка сервера `backend-2`

1. Подключиться к серверу `backend-2`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/sql-slave.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. Во время выполнения скрипта указать IP-адрес master-сервера.
5. Во время выполнения скрипта ввести пароль пользователя `repl`.
6. После завершения скрипта убедиться, что slave-сервер подключен к master и репликация работает корректно.

### Настройка базы данных для WordPress

1. Подключиться к серверу `backend-1`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/sql-wp.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. Во время выполнения скрипта задать пароль для пользователя `wordpress`.
5. После завершения скрипта убедиться, что база данных и пользователь созданы.

Для проверки репликации на сервере `backend-2` выполнить команду:

```bash
mysql -e "SHOW DATABASES;"
```

В списке должна отображаться база данных `wordpress`.

### Развертывание WordPress

WordPress должен быть развернут на обоих backend-серверах.

#### Сервер `backend-1`

1. Подключиться к серверу `backend-1`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/wordpress.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. Во время выполнения скрипта указать IP-адрес master-сервера и пароль пользователя `wordpress`.
5. После завершения скрипта проверить, что контейнер WordPress запущен:

```bash
docker ps -a
```

6. Проверить доступность локального web-сервера:

```bash
curl -s http://127.0.0.1:8080/health.txt
```

#### Сервер `backend-2`

1. Подключиться к серверу `backend-2`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/wordpress.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. Во время выполнения скрипта указать IP-адрес master-сервера и пароль пользователя `wordpress`.
5. После завершения скрипта проверить, что контейнер WordPress запущен:

```bash
docker ps -a
```

6. Проверить доступность локального web-сервера:

```bash
curl -s http://127.0.0.1:8080/health.txt
```

### Настройка дампа

1. Подключиться к серверу `backend-2`.
2. Скачать скрипт:

```bash
curl -O https://raw.githubusercontent.com/RanisAbit/otus/refs/heads/main/scripts/sqldump.sh
```

3. Сделать файл исполняемым и запустить скрипт.
4. После выполнения скрипта убедиться, что создан дамп базы данных и сохранена позиция бинарного лога.

По умолчанию позиция бинарного лога сохраняется в файл:

```bash
/tmp/sql_dump/binlog_position
```

## Проверка работоспособности

После завершения всех этапов необходимо убедиться, что:
- MySQL master/slave работает корректно;
- репликация между серверами активна;
- база данных `wordpress` существует на master и slave;
- контейнер WordPress запущен на обоих backend-серверах;
- `health.txt` доступен локально на каждом backend-сервере;


## Команды для проверки

Проверка контейнеров:

```bash
docker ps -a
```

Проверка списка баз данных:

```bash
mysql -e "SHOW DATABASES;"
```

Проверка доступности WordPress:

```bash
curl -s http://127.0.0.1:8080/health.txt
```

## Примечания

- В скриптах настройки MySQL уже используется пользователь `repl`; при запуске необходимо только задать пароль для данной учетной записи.
- В скрипте `sql-wp.sh` уже заданы имя базы данных и имя пользователя `wordpress`; необходимо указать только пароль.
- В рамках проекта используется  mysql-server-8.0 , который доступен по умолчанию в репозиториях Ubuntu в VK Cloud. При использовании других версий MySQL возможны различия в синтаксисе команд репликации: например, в MySQL 8.4 команда  SHOW MASTER STATUS  больше не поддерживается и заменена на  SHOW BINARY LOG STATUS .
- Часть конфигурационных файлов и Docker Compose-описаний формируется внутри скриптов через heredoc (`EOF`).
- При изменении логики развертывания необходимо вносить изменения не только в отдельные конфигурационные файлы репозитория, но и в соответствующие скрипты.
